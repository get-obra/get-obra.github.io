Agent Skills with Obra

How Obra implements the agentskills.io open standard: discovery, activation, execution, spec compliance, and where to put a SKILL.md folder.

Specagentskills.io/specification
Verified28 May 2026

Obra is a managed-service AI virtual employee for regulated SMBs — today focused on EU short-term-rental hosts who must register each arriving guest with their national identity registry (Portugal's SEF SIBA, Spain's SES Hospedajes, Italy's Polizia Hotelier, others).

We are also the first vertical-managed-service implementation of the agentskills.io open standard — joining a client list otherwise dominated by horizontal coding agents (Claude Code, Cursor, GitHub Copilot, VS Code, Codex, Gemini CLI, Goose, Letta, Hermes Agent, OpenHands, and 30+ more).

This page documents how Obra implements the standard: how skills are discovered, how the chat brain activates them, what the runtime executes, and where to put a skill folder if you want Obra to pick it up.

What a skill is, in one sentence

A skill is a folder with a SKILL.md file that teaches an AI agent how to do a specific task. Obra reads them as procedural knowledge — the chat brain knows when to use a skill from its description, then loads the full instructions when activated.

The full agentskills.io specification is at agentskills.io/specification. Obra's compliance is verified against the upstream skills-ref library via our adapter package.

Where Obra looks

After installation, Obra reads skills from <OBRA_HOME>/skills/ — typically ~/.obra/skills on Linux and macOS or %USERPROFILE%\.obra\skills on Windows. Each subdirectory is one skill; the directory name must match the name field in the skill's SKILL.md frontmatter (this is an agentskills.io spec requirement, enforced by Obra at load time).

~/.obra/
└── skills/
    ├── siba-checkin-fill/
    │   ├── SKILL.md          # required
    │   ├── scripts/          # optional
    │   ├── references/       # optional
    │   └── assets/           # optional
    ├── airbnb-reservation-extract/
    │   └── SKILL.md
    └── passport-mrz-verify/
        └── SKILL.md

How activation works — progressive disclosure

Obra implements the spec's three-stage discovery → activation → execution model.

Discovery — every chat turn

On every turn, the brain reads the name + description of every installed skill — roughly 100 tokens per skill. This block lives in the system prompt next to Obra's connector capability list, so the brain knows what procedural knowledge it has on hand without the host having to remind it.

If a host says "file this guest's passport with SIBA", the brain sees siba-checkin-fill in the discovery list and knows it has procedural knowledge for the task. If the request matches no skill, the brain replies from its own knowledge.

Activation — on relevance

When a request matches a skill's description, Obra's runtime loads the full SKILL.md body into the brain's next-turn context. The body's instructions guide the brain through the task.

You can also inspect the loaded state via Obra's local API:

GET http://localhost:3737/api/skills              # list all installed
GET http://localhost:3737/api/skills/<skill-name> # load full detail

Execution — on demand

When a skill's instructions reference bundled files (a scripts/extract.py or a references/REFERENCE.md), the brain reads them via its file-read tools at the moment they are needed. Resource paths are relative to the skill folder.

Spec compliance Obra enforces

Every skill goes through validation before it is surfaced in the chat brain's prompt. Skills that do not pass are silently skipped, so a broken skill folder never poisons your Home screen.

RuleEnforced
Folder contains SKILL.md✓
YAML frontmatter is parseable✓
name is 1-64 lowercase-alnum + hyphens; no leading/trailing/consecutive hyphens✓
name matches parent directory name✓
description is 1-1024 chars✓
Optional license, compatibility, metadata, allowed-tools✓

Soft warnings (do not block loading): description shorter than 40 chars (probably under-described) and body longer than 500 lines (consider splitting into references/).

Installing a skill into Obra

Three supported paths:

  1. Manual copy. For dev or personal use, drop a SKILL.md folder directly into the skills directory: cp -r my-skill ~/.obra/skills/. The next chat turn discovers it.
  2. Operator-pushed bundle (managed clients). When you are on a managed Obra deploy, your operator can publish a signed bundle of type skill through the capability-intake pipeline. The bundle lands at your machine, surfaces on Home as a "new capability awaiting your review" card, and installs into <OBRA_HOME>/skills/ after you accept at the PC. This is the trusted distribution path for vertical libraries Obra's operator maintains for your industry.
  3. Obra-shipped showcase skills. Obra publishes 3 MIT-licensed vertical reference skills: siba-checkin-fill (Portuguese SEF SIBA boletim filling), airbnb-reservation-extract (extract structured reservation data from Airbnb host emails), passport-mrz-verify (ICAO 9303 passport MRZ parse + checksum verification). Any of these can drop into a client's skills directory as-is, or load into any other agentskills.io-compatible agent.

Authoring a skill that targets Obra

Most agentskills.io-compatible skills work in Obra unchanged. A few Obra-specific notes:

  • Lean on description keywords. Obra's brain matches host requests against descriptions. Include both what the skill does and when to use it, with 2-3 keywords the host might naturally say.
  • Body should read as procedural knowledge. Step-by-step instructions work; narrative prose with no concrete steps doesn't.
  • References live one level deep. Big detail blocks belong in references/REFERENCE.md. Keep SKILL.md under 500 lines for fast activation.
  • compatibility is for environment requirements — "Designed for Obra managed runtime" is fine if the skill depends on Obra-specific connectors.
